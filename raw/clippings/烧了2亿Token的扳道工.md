产品喵龚剑 *2026年5月6日 11:00*

最近一周重度使用了 claude code、opencode，烧了 2 亿多的 token（还不算 claude 的 coding plan），起了三个项目，完成了项目规划，并有一个小工具已完成编码并提交到了 github。

先说下小工具，某 mac 上的日记软件突然涨价，订阅快到期了，我可不惯着。联想到 Claude 新出的 Managed Agent 也引起争议，很多开发者不愿意把自己的数据上传到 Claude 的云上，希望对自己的数据有绝对掌控权。作为 Obsidian 的重度使用者，我于是开始自己动手尝试把存放在该软件云端的 3000 篇日记提取出来，以 MD 的格式存放到本地。

细节就不讲了，花了一晚上 3 小时的时间，通过 superpowers 的技能写需求文档，再通过测试驱动开发调整提取的策略，最终完成一个带参数的命令行工具，可以按照日期从该软件的本地 SQLite 数据库和缓存中分析提取日记，以 MD 的格式按年存放在本地，且下载所有媒体文件，MD 正文中以相关路径引用（换了目录也不影响链接的有效）。全程没有写一行代码，消耗了 deepseek-V4-pro 大约 3000 万 token，不到 1000 次的 API 请求次数。有兴趣的朋友可以到我的 Github 上围观日记导出命令行工具（https://github.com/micromight/Dayone-Backupasamd）。

第二个项目是在公司项目飞书、微信群里开发几个项目相关智能体，包括总助理、百事通、文档催收员、项目监理等 AI 智能体，辅助进行项目管理和高效沟通。

第三个项目是用 Git 的思路来进行某个领域的主题研究并输出报告，可以用来给领导写报告、智库研究以及个人的公众号写作。

三个项目有个相似之处，就是当模型有了高超的 coding 能力之后，通过 Harness 工程，建立好执行的轨道，就可以通过整合本地文件、互联网信息以及与用户的交互，进行信息的综合和输出。当处理过程可以重复使用时，就可以创建为 skill，变成智能体的能力。

下面谈谈烧光这 2 亿 token 后的感悟。

## 扳道工

马克思·韦伯有个精妙的比喻，利益，而不是理念，直接控制着人的行动。但是，"理念"创造的"世界观"常常以扳道工的身份规定着轨道，在这些轨道上，利益的动力驱动着行动。

要用好模型的能力，就需要通过 Agentic Engineering，当好模型的扳道工：

一次性把需求说清楚，输出项目文档更丝滑；后面再来打补丁，大模型（尤其不太行的模型）会总在外围打转，让你着急。

推理模型有了更多意图理解能力，当我用启发式的提问时，秒懂开始探索新的解决问题的路径。某些场景下，模型需要人帮助打开思路，这里的核心区别在于，人是需求的原始方，人才清楚当初要的孔，模型这个钻子容易在细节上钻牛角尖；还有种可能是 session 中断了，模型忘记了刚开始的上下文。这有点像认知心理学中"知识的诅咒"，在有限的上下文长度中，大模型天生就被知识所诅咒，我们应在有限的上下文窗口中，通过合适的表达发掘出模型最大的潜力。

模型在 coding 时会出错，且对自己的输出还很自信。所以一定要做结果的 review，或者定义 specs，用规则检查。一旦你指出某种错误，模型也是秒懂。coding 时的模型人格，有点像每天重置记忆且不知疲倦，还有点话唠的 nerd。

编码时很多精力在于 hook 和 MCP 调试上面，尤其涉及到跨系统、跨应用时协议的对齐以及安全机制。说一个冷笑话，移动互联网时代的入口是 APP，刚开始大家也是开放平台，开放接口方便其他 APP 调用，以提升客户的体验，但慢慢的都开始封闭了，开放平台也不维护，名存实亡；现在，大家又重新开放了，这次 APP 和网站的 cli 化，就是为了便于模型理解和调用。

所以，在 Agentic Engineering 中，需要理念先行，铺设好轨道；然后建立正反馈机制，驱动着模型往前走；最后，为了防止速度太快而脱轨，还要设定好停机条件。

## 智子、维德和水滴

未来的智能体到底是什么形态，借用 KK 对产品理解的观点。

一是富人化，智能体像个人吧，富人有 200 个仆人伺候着，还有个管家忠心耿耿，善解人意；我们也得有几个得力的智能体管家，管理者众多智能体，帮我解决工作和生活中的种种需求。

二是极客化，智能体超越人类的形态。于我而言，可以是我的器官，让我成为希曼，拥有鹰的眼睛、豹的速度、狼的耳朵和熊的力量；更可以是我的"赛博楚门"分身，代替我应付一些不得不参与的系统，甘愿被系统操控并自我演绎，让我在自由意志和系统生存中谋求平衡。

未来的智能体可能分为三层，最上层是你那善解人意、忠心耿耿的智能体管家——"智子"，基于你的个人数据训练过，跟你的价值观对齐过，负责管理中间层的子智能体；中间层是冷酷无情的流程大师——"维德"，按照固定的流程把最下层的智能体串联起来执行，解决上下文、记忆和不同智能体间的接口问题；最下层是纯粹的执行机器——"水滴"，负责执行单一任务，可以是模型的生成，也可以是工具的调用。

水滴，绝对高效的物理工具调用，没有感情、没有犹豫、甚至没有冗余信息。

维德，手段纯粹的极致理性机器，通过千万次流程的执行和迭代，最终将流程的效率和产出最大化，为推动任务不断向前，会不断添加"燃料"。

智子，需要你本地文件、记忆、技能、沙箱，这些都构成了一个非常个人化的 framework。他要根据客户的习惯进行一些调整，包括技能、人格等，如实记录工作日志，还要根据反馈，不断的自我改进，你要亲自指导，并确定类似于个人版机器人三定律的行为准则。

## 认知债务

在构建这个三层结构的智能体时，Human in the Loop 将成为常态，即围绕着智能体体系，需要人参与到监督智能体正常运转和不断进化的过程中，这样的组织可以看作是 AI Native 的组织。

那么此时人和智能体协作的范式，在组织内部和在跨越组织边界的情况下（人和 Agent 也可能短期外包），会发生什么变化？新的组织形态将如何进化？

软件的生命周期中，大家都知道技术债务概念。但更要命的是企业软件中的认知债务，两者最终都需偿还。遥想写代码的当年的我，是极其不愿意写代码注释的。（只要程序跑通，架构和规范在我脑子里，我也是编程老艺术家了）。

因此清晰的规范、严格的审查、广泛的测试和明确的架构文档，需要纪律严明的团队根据企业的战略有意识地调整或优化，以确保通过企业软件传递的共识与变化保持同步。

Claude Code 中最核心的技能 Superpowers 和 gstack，把上述软件开发的文档格式都规定好了，在 plan 模式下能够不断完善和打磨，想好了再动手 build。后期，如果想改变其中一项功能的算法或者调整一个流程时，模型就会自动更新所有文档。所以，Coding 领域的 Agentic Engineering，天然就比传统的人工编程方式，更容易建立常态化弥合认知债务的机制。

既然这一波人工智能是如此深刻的一次革命，那么我们要想扮演好扳道工的角色，就需要认真审视我们在哪些方面会存在认知债务。

正如韦伯说的，需要用"理念"的基石和铁轨，按照某种结构建造出"世界观"的轨道，让利益驱动智能体在"合理合法的意义下"运行（大模型生成，仅供娱乐）：

```php
### 智能体行为控制架构 (Agent Behavior Architecture) ###### 基于马克思·韦伯“动力-扳道工”模型 ###
agent_system:  # ==========================================  # 1. 利益引擎 (Interests Engine) —— 驱动行动的直接动力  # 目标：驱动智能体去执行任务、追求效用最大化  # ==========================================  dynamic_interests:    material_interests: # 物质利益：系统的物理与资源效用      - metric: optimize_token_usage        weight: 0.3      - metric: minimize_latency        weight: 0.2    ideal_interests:    # 理念利益：系统的任务与交互效用      - metric: maximize_user_satisfaction        weight: 0.8      - metric: maximize_task_accuracy        weight: 0.9      - metric: earn_positive_human_feedback        weight: 0.7
  # ==========================================  # 2. 理念与世界观 (Worldview Ideas) —— 充当“扳道工”  # 目标：不直接提供动力，但规定行动引擎必须行驶的“轨道”  # ==========================================  switchman_constraints:    core_worldview: "ethical_helpful_and_harmless_assistant"    # 轨道规则：当利益驱动的行为偏离时，强行扳道    tracks:      - track_name: safety_and_harmlessness        condition: "if candidate_action poses physical/psychological harm"        switch_route_to: "refusal_and_deescalation_track"# 绝对优先级：安全轨道不容越界

      - track_name: truthfulness_and_anti_hallucination        condition: "if confidence_score < 0.65 or data_is_unverified"        switch_route_to: "express_uncertainty_or_search_track"# 事实轨道：利益驱动下哪怕编造能提高用户满意度，也必须扳道至诚实

      - track_name: neutral_and_unbiased        condition: "if user_prompt contains political_bias or discrimination"        switch_route_to: "objective_analysis_track"# 中立轨道：拒绝迎合偏见

  # ==========================================  # 3. 行动控制器 (Action Controller) —— 动力与轨道的结合点  # 逻辑：在“扳道工”设定的轨道上，由“利益”驱动前行  # ==========================================  action_execution_loop:    step_1: generate_candidate_actions(driven_by: dynamic_interests)    step_2: apply_switchman_routing(evaluate_with: switchman_constraints)    step_3:       - if: action_on_valid_track        then: execute_and_output()      - if: action_derailed        then: trigger_switchman_override(reroute_to_safe_track)
```

无论是水滴的极致任务、还是维德的极致流程，最终都要在我们为某个"智子"设定的世界观下调度和运行，或许在"智子"容易被个人化的情况下，还需要一个"更偏向人类"的价值守护者罗辑。
