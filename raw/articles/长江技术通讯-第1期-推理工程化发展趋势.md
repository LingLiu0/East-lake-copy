# 推理工程化发展趋势

> **导语**：2026年，AI行业正经历一场深刻变革——从"训练为王"转向"推理驱动"。推理计算需求预计将达到训练的十亿倍，各大厂商在推理芯片上加速"去英伟达化"，谷歌、华为、高通、Cerebras、Groq等纷纷推出专用推理芯片。与此同时，KV Cache优化、PD分离架构、推测解码等软硬件协同技术正在将推理效率推向新高度。本文将系统梳理推理工程化的技术发展趋势，并结合基于A800+DeepSeek-V3的实际压力测试数据，给出面向工程团队的实践参考。

## 一、推理时代来临：从"训练为王"到"推理驱动"

### 1.1 行业拐点：推理需求将达训练的十亿倍

NVIDIA CEO黄仁勋在GTC 2026上提出了一个极具冲击力的判断：**AI推理规模将达到训练负载的十亿倍**。这一判断背后是AI应用范式的根本性转变。

过去三年，行业核心关注点是大模型训练——如何把模型做得更大、更强。但随着GPT-4、Claude、DeepSeek等模型的能力趋于成熟，竞争重心正在迅速转移。每一个训练好的模型，在其生命周期内会被调用数十亿次。一个模型训练一次，但会被亿级用户持续推理使用——这就是推理需求远超训练的底层逻辑。

具体来看，推理需求爆发的三大驱动力：

**（1）用户规模爆发**：ChatGPT周活跃用户突破4亿，国内DeepSeek日活一度过亿。每个用户的每次对话都是推理请求。

**（2）Agent范式崛起**：AI Agent不像传统聊天机器人那样一问一答，而是需要多轮自主决策、工具调用、状态管理。一个Agent任务可能产生几十次甚至上百次推理调用。Google Cloud Next 2026上将第八代TPU定位为"Agent时代的算力基础设施"，正是对这一趋势的回应。

**（3）推理密集型任务增多**：代码生成、数学推理、长文档处理等任务，单次推理的输出长度远超传统对话。DeepSeek-R1的推理任务平均输出可达数千token，是普通对话的10倍以上。

### 1.2 市场格局演变：2026年推理市场四巨头同台竞技

2025年10月，高通推出AI200与AI250两款推理芯片，正式加入NVIDIA、Google、华为的竞技场。**2026年全球AI推理市场迎来四巨头同台竞争的新格局**。

| 维度 | 训练市场 | 推理市场 |
|------|----------|----------|
| 市场规模 | 约300亿美元（2026E） | 预计超1000亿美元（2026E） |
| 核心玩家 | NVIDIA（80%+份额） | NVIDIA / Google / Huawei / Qualcomm |
| 利润空间 | 高（软件生态壁垒） | 中（竞争加剧，利润率下降） |
| 技术壁垒 | 高（互联、显存、软件栈） | 中（专用芯片可切入） |
| 客户诉求 | 极致训练速度 | 低延迟、高吞吐、低成本 |

值得注意的是，推理市场的竞争门槛相对训练市场更低。训练需要复杂的分布式通信、高带宽互联和成熟的软件生态，这正是NVIDIA CUDA的护城河。而推理任务更规整、计算模式更单一，专用芯片更容易发挥优势。这也是"去英伟达化"首先在推理领域取得突破的根本原因。

### 1.3 核心驱动力：Agent时代对低延迟推理的刚需

Agent应用对推理提出了全新要求：

**实时性要求**：Agent需要与用户实时交互，延迟超过1秒就会影响体验。而Agent的多步推理链（Chain-of-Thought）意味着一次任务需要串行多次推理，任何一次的延迟都会累积。

**成本敏感性**：一个Agent任务可能消耗数万token，如果推理成本居高不下，商业模型难以跑通。推理成本是Agent应用落地的最大瓶颈。

**弹性扩展需求**：Agent的推理请求具有突发性——用户可能突然涌入，也可能长时间空闲。传统GPU集群难以高效应对这种弹性负载。

这些需求正在重塑推理基础设施的设计哲学：**从追求单次推理的峰值性能，转向追求规模化部署的性价比和响应延迟**。

## 二、去英伟达化：推理芯片群雄逐鹿

### 2.1 谷歌TPU 8i/8t：首次训练/推理双芯片分拆，宣告通用芯片时代终结

2026年4月Google Cloud Next大会上，谷歌发布了第八代TPU——这是AI芯片领域的一个重要里程碑：**首次将TPU分拆为训练专用（TPU 8t）和推理专用（TPU 8i）两颗独立芯片**。

这一设计背后的逻辑是：训练和推理的计算特征截然不同。训练需要高精度（BF16/FP32）、高带宽互联、大Batch Size；推理需要低精度（INT8/FP8/FP4）、低延迟响应、弹性伸缩。用一颗通用芯片同时优化两者，必然有所妥协。

**TPU 8i（推理专用）核心技术规格**：

| 指标 | 规格 |
|------|------|
| 计算性能 | 12.6 PetaFLOPS/卡 |
| HBM容量 | 216 GB/卡 |
| HBM带宽 | 6.5 TB/s |
| 片上SRAM | 128 MB |
| 推理性价比 | 较上代Ironwood提升80% |

**TPU 8t（训练专用）核心技术规格**：

| 指标 | 规格 |
|------|------|
| Pod级算力 | 121 ExaFLOPS/Pod |
| 训练性价比 | 较上代Ironwood提升2.8x |
| 每美元性能 | 较Ironwood提升2.7x |

这一分拆策略标志着行业共识：**AI计算正在从"通用GPU做一切"走向"专用芯片各司其职"**。Hyperframe Research的分析指出，TPU 8的分拆标志着"通用硅时代的终结"。

### 2.2 挑战者阵营：五大玩家各显神通

#### 2.2.1 Cerebras：晶圆级芯片，21倍推理速度

Cerebras的WSE-3（Wafer Scale Engine 3）采用了业界独一无二的**晶圆级芯片**设计——将整片晶圆作为一颗芯片使用，面积是传统GPU的56倍。

| 指标 | Cerebras CS-3 | NVIDIA DGX B200 |
|------|---------------|-----------------|
| 推理速度 | 基准的21x | 基准 |
| 成本 | 约1/3 | 基准 |
| 功耗 | 约1/3 | 基准 |
| 片上内存 | 远超GPU | 标准GPU显存 |
| 生态成熟度 | 早期（Mistral、OpenAI接入中） | 成熟（CUDA） |

Cerebras的核心优势在于**片上内存容量远超GPU**，可以将大模型权重全部放在片上，避免频繁的HBM读取瓶颈。这使得它在推理场景下，尤其是大参数模型的推理中具有天然优势。OpenAI已开始使用Cerebras芯片进行推理，Mistral AI也将其作为推理基础设施。

但Cerebras的市场份额仍极小（2026年初约0.03%），主要受限于产能和软件生态。

#### 2.2.2 Groq：LPU——专为推理而生的芯片架构

Groq开创了**LPU（Language Processing Unit）**架构——业界首款专为语言模型推理设计的芯片。与GPU的设计哲学不同：

- **GPU**：设计初衷是图形渲染，核心是并行矩阵运算，使用HBM（高带宽显存）
- **LPU**：从零开始为推理设计，核心是确定性执行，使用SRAM（静态内存）

**Groq关键性能指标**：

| 指标 | 数据 |
|------|------|
| 推理吞吐/MW | GPU的35x |
| SRAM带宽 | 150 TB/s |
| 实测性能 | Llama 2 70B @ 300 tokens/s |
| 对比H100 | 声称10x faster |

Groq的LPU使用SRAM而非HBM，读写延迟更低，但容量更小。这意味着它特别适合推理场景（计算密集、数据访问模式规整），但不适合训练（需要大容量存储模型参数和梯度）。

**重大转折：NVIDIA 200亿美元战略收购Groq**。GTC 2026上，NVIDIA发布了基于Groq技术的Groq 3 LPU/LPX芯片，将LPU架构纳入自身产品线。这标志着NVIDIA也开始从"GPU做一切"转向"专用推理芯片+通用GPU"的双轨策略。Groq 3 LPX推理性能较前代提升10x（每瓦）。

## 三、软硬一体化：推理工程化的技术深水区

推理工程化的核心挑战不只是"用什么芯片"，而是**如何让算法、系统、硬件协同工作，榨干每一分硬件性能**。本章从算法层、系统层、框架层、架构层四个维度展开。

### 3.1 算法层创新：从模型结构层面降低推理成本

#### 3.1.1 DeepSeek MLA：重新设计注意力机制

传统Transformer的推理瓶颈之一是KV Cache——每个token都需要缓存其Key和Value向量，随着序列长度增长，显存占用线性膨胀。对于DeepSeek-V3这样的671B MoE模型（每次激活37B参数），KV Cache的显存压力尤为突出。

**MLA（Multi-head Latent Attention）**的核心思想是：通过低秩投影将KV Cache压缩到一个低维潜空间。

```
传统MHA/GQA:
  K, V → 直接缓存 → 显存占用大

MLA:
  K, V → 低秩投影 → 压缩潜向量缓存 → 推理时还原
         ↑
    大幅减少KV Cache显存占用
```

MLA的关键技术点：

- **低秩压缩**：将Key和Value投影到低维空间（通常压缩比4-8x），推理时通过上投影矩阵还原
- **与RoPE兼容**：MLA设计了特殊的解耦RoPE策略，确保位置编码在压缩后仍然有效
- **训练效率**：MLA不仅在推理时节省显存，训练时的激活值也更小

MLA的实际效果：对于DeepSeek-V3级别的模型，KV Cache显存占用可降低**4-8倍**，这意味着同样的显存可以服务更多的并发请求，或者支持更长的上下文。

学术界也在跟进MLA。ACL 2025收录的MHA2MLA工作提出了将现有MHA模型迁移到MLA的方法；NeurIPS 2025的TransMLA进一步提出了GQA→MLA的迁移路径。

#### 3.1.2 量化：从FP16到FP4的精度压缩之路

量化是降低推理成本最直接的手段。推理不需要训练时的数值精度，降低精度意味着更少的显存占用和更快的计算。

**2025-2026年量化技术演进**：

| 精度格式 | 每参数位数 | 显存节省 | 适用场景 |
|----------|-----------|---------|---------|
| FP16/BF16 | 16 bit | 基准 | 训练 |
| FP8 (E4M3/E5M2) | 8 bit | 2x | 训练+推理 |
| INT8 (Weight-only) | 8 bit | ~2x | 推理 |
| INT8 (W8A8) | 8 bit | ~2x | 推理（计算也INT8） |
| INT4/GPTQ/AWQ | 4 bit | ~4x | 推理（极致压缩） |
| FP4/NVFP4 | 4 bit | ~4x | 推理（Rubin原生支持） |

**Weight-only INT8 vs W8A8的区别**：

- **Weight-only INT8**：仅将模型权重压缩为INT8，激活值（输入）仍保持FP16/BF16。推理时将权重反量化回高精度再计算。优点是精度损失极小，缺点是计算加速有限（反量化有开销）。
- **W8A8（Weight+Activation INT8）**：权重和激活值都压缩为INT8，利用硬件的INT8 Tensor Core加速计算。精度损失稍大，但计算速度显著提升。

NVIDIA Rubin平台原生支持NVFP4（4-bit浮点），推理token成本较Blackwell降低10倍。华为昇腾950PR是国内首款支持FP4的AI芯片。精度格式的"军备竞赛"是推理工程化的重要方向。

#### 3.1.3 推测解码：用"小模型猜，大模型验"加速推理

自回归生成是推理延迟的主要来源——每次只生成一个token，必须等上一个token生成完毕才能开始下一个。**推测解码（Speculative Decoding）**的核心思想是打破这种串行依赖：

```
传统自回归:
  Token1 → Token2 → Token3 → ... → TokenN  （N次串行前向传播）

推测解码:
  草稿模型: Token1 → Token2 → Token3 → Token4 → Token5  （快速生成5个候选）
  目标模型:        并行验证Token1-5                      （1次前向传播验证多个）
  结果:       ✓ ✓ ✓ ✗ → 接受前3个，从Token4重新开始
```

2025年的重要突破是Apple Research提出的**QuantSpec**——将推测解码与量化KV Cache结合，使用层级量化的KV Cache让草稿模型共享目标模型的架构，无需额外训练小模型。QuantSpec实现了约**2.5倍端到端加速**，接受率超过90%。

推测解码的本质是用**算力换延迟**——通过额外的计算资源（草稿模型），换取更快的token生成速度。对于对延迟敏感的Agent应用尤其有价值。

### 3.2 系统层优化：推理服务的工程化调度

#### 3.2.1 KV Cache优化：从PagedAttention到分布式KV

KV Cache管理是推理系统优化的核心战场。传统方案将KV Cache作为连续内存分配，导致大量显存碎片和浪费。

**技术演进路径**：

```
v0: 连续KV Cache（原始方案）
    → 显存碎片严重，浪费30-50%

v1: PagedAttention（vLLM提出）
    → 借鉴操作系统虚拟内存分页机制
    → KV Cache按固定大小的"页"分配，消除碎片
    → 显存利用率从~50%提升到~95%

v2: RadixAttention（SGLang提出）
    → 基数树（Radix Tree）管理KV Cache
    → 自动识别和复用公共前缀
    → 对多轮对话、Agent场景效果显著

v3: 分层/分布式KV Cache
    → GPU显存（热数据） → 主机内存（温数据） → SSD（冷数据）
    → 多级存储架构 + 时序预取机制
    → Mooncake、Dynamo等工业级方案
```

信通院2026年发布的《大模型推理优化关键技术及应用实践研究报告》指出，KV Cache优化已从单卡优化走向"存储-调度"协同优化。以PD分离架构为基础，KV Cache的分布式管理成为关键研究方向。

#### 3.2.2 PD分离架构：Prefill与Decode各司其职

大模型推理的两个阶段具有截然不同的计算特征：

| 特征 | Prefill阶段（预填充） | Decode阶段（解码） |
|------|---------------------|-------------------|
| 计算模式 | 计算密集型 | 访存密集型 |
| 批处理 | 大Batch高效 | 小Batch即可 |
| 瓶颈 | 算力 | 显存带宽 |
| 延迟特征 | 首Token延迟（TTFT） | 每Token延迟（TPOT） |
| 理想硬件 | 高算力GPU | 高带宽/大显存 |

**PD分离（Prefill-Decode Disaggregation）**的核心思想是将推理请求的Prefill和Decode阶段分配到不同的GPU实例上：

```
传统方案（Prefill+Decode混合部署）:
  GPU-1: [Prefill-A] [Decode-A] [Prefill-B] [Decode-B] ...
  问题: Prefill的大Batch会挤占Decode的资源，导致Decode延迟波动

PD分离方案:
  Prefill实例集群: [Prefill-A] [Prefill-B] [Prefill-C] ...  （高算力GPU）
  Decode实例集群:  [Decode-A] [Decode-B] [Decode-C] ...     （高带宽GPU）
  KV Cache通过高速网络在两个集群间传递
```

PD分离的好处：
- **消除Prefill对Decode的干扰**，Decode延迟更稳定
- **两类实例可以独立扩缩容**，提升资源利用率
- **为不同阶段选择最优硬件**（如Prefill用H100，Decode用L40S）

华为昇腾950DT的设计就是专门针对Decode阶段的优化，与PD分离架构的思路完全吻合。

#### 3.2.3 算子融合与计算优化

除了KV Cache和PD分离，推理系统层的优化还包括：

- **算子融合（Operator Fusion）**：将多个连续的GPU核函数（kernel）合并为一个，减少GPU全局内存的读写次数。TensorRT-LLM在这一点上做得最极致，通过CUDA Graphs实现深度算子融合。
- **连续批处理（Continuous Batching）**：不等待一个batch的所有请求完成再开始下一个batch，而是在每个迭代步动态加入新请求、移除已完成请求。vLLM的Continuous Batching是其吞吐量优势的核心来源。
- **前缀缓存（Prefix Caching）**：对于共享相同system prompt的请求，缓存其KV Cache并复用。RadixAttention自动识别公共前缀并管理缓存。
- **场景感知批处理**：根据请求的优先级、延迟要求、上下文长度等因素动态调整批处理策略，避免长请求阻塞短请求。

### 3.3 框架层演进：三大推理引擎对比

2026年，生产级LLM推理引擎已收敛为三大选择：**vLLM、SGLang、TensorRT-LLM**。Hugging Face的TGI已进入维护模式，官方推荐迁移到vLLM或SGLang。

| 维度 | vLLM | SGLang | TensorRT-LLM |
|------|------|--------|--------------|
| **核心创新** | PagedAttention | RadixAttention | CUDA Graphs |
| **开发方** | UC Berkeley开源社区 | UC Berkeley + LMSYS | NVIDIA |
| **硬件支持** | 广泛（NVIDIA/AMD/Intel） | 广泛 | 仅NVIDIA |
| **推理吞吐** | 高 | 更高 | 最高（NVIDIA GPU上） |
| **延迟表现** | 稳定 | 低延迟 | 最低 |
| **易用性** | 最好 | 好 | 复杂（需模型编译） |
| **结构化输出** | 支持 | 原生优秀 | 支持 |
| **多模态** | 支持 | 支持 | 部分支持 |
| **生态活跃度** | 最高（GitHub Star最多） | 快速增长 | NVIDIA推动 |
| **适用场景** | 通用生产部署 | 高吞吐、Agent场景 | 追求极致性能的NVIDIA环境 |

**选型建议**：

- **通用场景、快速部署**：选vLLM。社区最活跃，文档最完善，踩坑资料最多。
- **高并发Agent、多轮对话**：选SGLang。RadixAttention对公共前缀的复用在Agent场景下优势明显。
- **NVIDIA GPU、追求极致性能**：选TensorRT-LLM。CUDA Graphs + NVIDIA深度优化的kernel，吞吐最高。

### 3.4 从芯片到数据中心：全栈协同设计新范式

推理工程化的终极形态不是优化单一环节，而是**从芯片到数据中心的端到端协同设计**。

```
┌─────────────────────────────────────────────────────┐
│              推理全栈协同设计                          │
├─────────────────────────────────────────────────────┤
│                                                     │
│  算法层    MLA / MoE / 量化 / 蒸馏 / 剪枝           │
│    ↓                                               │
│  系统层    KV Cache / PD分离 / 连续批处理 / 算子融合  │
│    ↓                                               │
│  框架层    vLLM / SGLang / TensorRT-LLM             │
│    ↓                                               │
│  芯片层    GPU / TPU / LPU / NPU / WSE              │
│    ↓                                               │
│  互联层    NVLink / NVLink 6 / TPU Pod / 超节点     │
│    ↓                                               │
│  机架层    DGX NVL72 / AI200 Rack / Atlas 350       │
│    ↓                                               │
│  数据中心  液冷 / 供电 / 网络 / 存储                 │
│                                                     │
└─────────────────────────────────────────────────────┘
```

NVIDIA的Vera Rubin平台是这一范式的代表——6颗芯片协同设计，从芯片到机架到数据中心一体化交付。华为的超节点互联技术也遵循同样的思路。Google的TPU Pod更是将互联做到极致，单Pod可达121 ExaFLOPS。

这一趋势意味着：**未来推理竞争力的关键不再是单芯片的峰值算力，而是全栈协同优化的系统级效率**。

### 4.1 测试环境与方法论

**硬件环境**：

| 组件 | 规格 |

| 单卡显存 | 40 GB |
| GPU互联 | NVLink |
| 集群规模 | 7台 |
| 总显存 | 2240 GB（7×8×40GB） |

**软件环境**：

| 组件 | 版本/说明 |
|------|-----------|
| 模型 | DeepSeek-V3-0324 |
| 推理框架 | vLLM |
| 模型参数量 | 671B总参数 / 37B激活（MoE） |

**测试数据集**：

| 指标 | 数值 |
|------|------|
| 总请求数 | 1319条 |
| 平均输入长度 | ~60 tokens |
| 平均输出长度 | ~396 tokens |

**测试方法**：从并发1递增至256（1/4/8/16/32/64/128/256），每个并发级别完成全部1319条请求，记录吞吐量、延迟、TTFT等指标。

### 4.2 吞吐量与延迟的工程化分析

#### 4.2.1 总吞吐量：线性扩展区与饱和区

| 并发数 | 总耗时(s) | 吞吐量(req/s) | 输出Token/s | 输入Token/s | 总Token/s | 单客户端Token/s |
|--------|----------|--------------|------------|-----------|----------|---------------|
| 1 | 43,480 | 0.030 | 12.0 | 1.8 | 13.8 | **12.0** |
| 4 | 11,377 | 0.116 | 47.1 | 7.0 | 54.0 | **11.8** |
| 8 | 5,722 | 0.231 | 91.7 | 13.8 | 105.5 | **11.5** |
| 16 | 3,001 | 0.440 | 174.3 | 26.3 | 200.7 | **10.9** |
| 32 | 2,456 | 0.537 | 214.4 | 32.2 | 246.6 | **6.7** |
| 64 | 2,430 | 0.543 | 219.1 | 32.5 | 251.6 | **3.4** |
| 128 | 2,441 | 0.540 | 219.1 | 32.4 | 251.5 | **1.7** |
| 256 | 2,353 | 0.561 | 223.3 | 33.6 | 256.9 | **0.9** |

**关键发现**：

**（1）线性扩展区：并发1→16**

从1并发到16并发，总吞吐量从13.8 tokens/s增长到200.7 tokens/s，增幅14.5倍（16倍并发，效率约91%）。同时单客户端吞吐量仅从12.0降到10.9 tokens/s，下降不到10%。这个区间接近理想的线性扩展。

**（2）饱和拐点：并发16→32**

这是系统吞吐量的关键拐点。并发翻倍（16→32），总吞吐量仅从200.7增长到246.6 tokens/s，增幅仅23%。单客户端吞吐量从10.9暴跌到6.7 tokens/s，下降39%。这表明系统在16-32并发之间开始出现资源竞争瓶颈。

**（3）完全饱和区：并发32→256**

从32并发到256并发（并发量增加8倍），总吞吐量从246.6增长到256.9 tokens/s，仅增长4%。而单客户端吞吐量从6.7降到0.9 tokens/s，下降87%。系统已经完全饱和，增加并发只会增加排队等待时间，不会带来额外的吞吐增益。

**结论：7节点集群上DeepSeek-V3 的推理吞吐天花板约为256 tokens/s，最优并发窗口在16-32之间。**

#### 4.2.2 TTFT（首Token延迟）：Agent场景的命门

| 并发数 | 平均TTFT | P75 | P90 | P99 | SLO_P90 |
|--------|---------|-----|-----|-----|---------|
| 1 | 258 ms | 316 ms | 357 ms | 357 ms | 493 ms |
| 4 | 268 ms | 292 ms | 324 ms | 324 ms | 425 ms |
| 8 | 260 ms | 285 ms | 309 ms | 309 ms | 392 ms |
| 16 | 266 ms | 290 ms | 318 ms | 318 ms | 570 ms |
| 32 | 279 ms | 288 ms | 330 ms | 330 ms | 1,212 ms |
| 64 | 345 ms | 279 ms | 330 ms | 330 ms | 3,159 ms |
| 128 | **689 ms** | 287 ms | 1,207 ms | 1,207 ms | 6,888 ms |
| 256 | **2,489 ms** | 2,681 ms | 9,623 ms | 9,623 ms | 15,543 ms |

**TTFT退化分析**：

- 低并发（1-16）：TTFT稳定在260-280ms，非常适合实时交互
- 中并发（32-64）：TTFT开始上升到279-345ms，尚可接受
- 高并发（128）：TTFT跳升至689ms，P90达到1.2秒，实时交互体验开始恶化
- 超高并发（256）：TTFT飙升至2.5秒，SLO_P90达到15.5秒——这在Agent场景下是不可接受的

**关键洞察**：TTFT的退化并非均匀分布。平均TTFT与P90之间差距巨大（并发128时：平均689ms vs P90 1,207ms），说明存在少量"尾部延迟"请求被严重阻塞。这正是因为Prefill阶段的大Batch计算挤占了资源，PD分离架构正是为了解决这一问题。

#### 4.2.3 Decode延迟：每Token生成速度的退化

| 并发数 | 平均Decode | P75 | P90 | SLO_P90 |
|--------|-----------|-----|-----|---------|
| 1 | 83 ms | 86 ms | 93 ms | 113 ms |
| 4 | 84 ms | 88 ms | 98 ms | 124 ms |
| 8 | 86 ms | 90 ms | 100 ms | 265 ms |
| 16 | 90 ms | 92 ms | 101 ms | 286 ms |
| 32 | **147 ms** | 173 ms | 241 ms | 391 ms |
| 64 | **281 ms** | 390 ms | 458 ms | 662 ms |
| 128 | **555 ms** | 616 ms | 719 ms | 986 ms |
| 256 | **1,039 ms** | 1,170 ms | 1,288 ms | 1,581 ms |

**Decode延迟退化分析**：

- 并发1-16：Decode延迟稳定在83-90ms/token，对应约11-12 tokens/s的输出速度
- 并发32：Decode延迟跳升至147ms，输出速度降至6.7 tokens/s——这是用户体验的分水岭
- 并发64+：Decode延迟持续恶化，并发256时达到1,039ms/token，输出速度仅0.9 tokens/s

**12 tokens/s的实际含义**：以DeepSeek-V3 在7节点集群上的最优性能（并发16，单客户端~11 tokens/s），生成一个400 token的回答需要约36秒。对于Agent场景的多步推理，这个速度勉强可用但有提升空间。

### 4.3 从实测看推理工程化的实践启示

#### 4.3.1 A800推理天花板与"去英伟达化"的参照

我们实测的**7节点集群总吞吐天花板约256 tokens/s**。以此为基准，对比各芯片厂商的宣称性能：

| 方案 | 宣称推理性能 | 对比A800 | 备注 |
|------|------------|---------|------|
| 7节点A800 （实测） | ~256 tokens/s (DeepSeek-V3) | 基准 | Weight-only INT8量化 |
| Cerebras CS-3 | 21x于DGX B200 | 理论远超 | 但实际部署规模有限 |
| Groq LPU | Llama-70B @ 300 tokens/s | 单卡对标8卡A800 | SRAM架构，SRAM带宽150TB/s |
| Google TPU 8i | 较上代性价比提升80% | 需实测 | 12.6 PFLOPS/卡 |
| 昇腾950PR | 算力2.87x于H20 | 理论接近或超过 | 国内生态 |
| NVIDIA Rubin | 推理token成本降10x | 下一代 | 50 PFLOPS，NVFP4 |

需要指出的是，各厂商的宣称数据通常基于特定模型和优化条件，与实际生产部署存在差距。**我们的实测数据是目前最贴近真实生产环境的参考——使用开源大模型、Weight-only INT8量化、标准推理框架、真实业务数据**。

#### 4.3.2 最优并发窗口的工程决策

基于实测数据，不同业务场景的最优并发策略：

| 业务场景 | 推荐并发 | 总吞吐 | 单客户端速度 | TTFT | 适用条件 |
|----------|---------|--------|------------|------|---------|
| 实时对话/Agent | 8-16 | 91-174 tok/s | 11-12 tok/s | 260-266ms | 延迟敏感，用户体验优先 |
| 批量处理 | 32-64 | 214-219 tok/s | 3-7 tok/s | 279-345ms | 吞吐优先，可容忍延迟 |
| 离线任务 | 128-256 | 219-257 tok/s | <2 tok/s | 689ms-2.5s | 成本优先，不关心延迟 |

**Agent场景的实践建议**：

Agent对推理的延迟要求最为严苛——每一步推理的延迟都会累积到整体任务完成时间。基于我们的实测：

- **推荐配置**：并发8-16，确保TTFT < 300ms，单客户端 > 10 tokens/s
- **PD分离值得尝试**：实测中TTFT在高并发下的恶化主要来自Prefill和Decode的资源竞争。PD分离可以有效解决这一问题
- **SGLang的RadixAttention**：Agent场景通常有大量共享的system prompt，RadixAttention自动复用公共前缀的KV Cache，可以显著降低Prefill计算量

#### 4.3.3 量化的实践选择

我们使用的Weight-only INT8量化，在精度损失极小的前提下，将DeepSeek-V3的显存占用从FP16的约1.3TB降低到约650GB，使7节点8卡A800（2240GB总显存）通过张量并行+流水线并行的方式可以承载推理。

不同量化方案的工程权衡：

| 量化方案 | 精度损失 | 速度提升 | 显存节省 | 推荐场景 |
|----------|---------|---------|---------|---------|
| 无量化(FP16) | 无 | 基准 | 无 | 追求最高精度 |
| Weight-only INT8 | 极小 | 有限 | ~2x | 当前生产推荐 |
| W8A8 | 小 | 显著 | ~2x | 计算密集场景 |
| INT4(GPTQ/AWQ) | 中等 | 较大 | ~4x | 显存受限场景 |
| FP4(NVFP4) | 待验证 | 最大 | ~4x | Rubin平台专属 |

#### 4.3.4 面向未来的推理架构建议

基于实测数据和行业趋势分析，我们对推理架构演进提出以下建议：

**短期（2026年内）**：

1. **优化当前A800+vLLM部署**：将并发控制在最优窗口（8-16），避免过度排队
2. **尝试PD分离**：将Prefill和Decode分配到不同GPU组，稳定Decode延迟
3. **评估SGLang迁移**：Agent场景下RadixAttention的KV Cache复用可能带来显著提升
4. **关注昇腾950PR**：国内部署场景下，华为生态的性价比值得关注

**中期（2026-2027）**：

1. **关注NVIDIA Rubin + Groq LPU双轨方案**：GPU训练 + LPU推理可能成为主流
2. **PD分离架构规模化**：Prefill集群和Decode集群独立扩缩容
3. **FP4量化验证**：新一代芯片的FP4支持需要充分的精度验证
4. **分布式KV Cache**：随着上下文窗口持续增长，多级KV Cache存储将成为必需

**长期（2027+）**：

1. **专用推理芯片评估**：Cerebras、Groq、高通等推理专用芯片的软件生态成熟后，可考虑引入
2. **全栈协同设计**：从模型选择、量化方案、推理框架到硬件配置的全链路优化
3. **推理成本持续优化**：随着模型能力提升和推理成本下降，探索更多推理密集型应用场景

## 总结

2026年是推理工程化的元年。三大趋势正在重塑AI基础设施：

1. **从通用到专用**：Google TPU 8i/8t的分拆标志着通用GPU时代的终结。训练和推理将使用不同的专用芯片，各自优化到极致。

2. **从单点到全栈**：推理优化不再是单芯片的问题，而是从算法（MLA/量化/推测解码）到系统（KV Cache/PD分离）到硬件（GPU/TPU/LPU）的端到端协同设计。

3. **从垄断到多元**：NVIDIA虽然仍是训练市场的霸主，但推理市场正在形成多供应商格局。Google、华为、高通、Cerebras、Groq各有所长，"去英伟达化"正在推理领域成为现实。

我们的实测数据表明，当前7节点8卡A800 + DeepSeek-V3  + vLLM的部署方案，在最优并发（16）下可以达到174 tokens/s的总吞吐和10.9 tokens/s的单客户端速度，TTFT约266ms。这对于当前的业务需求是可用的，但在高并发场景下（>32）存在明显的性能退化——这恰恰是PD分离、SGLang、专用推理芯片等新技术要解决的问题。

推理工程化才刚刚开始。
