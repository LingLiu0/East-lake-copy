## 二、核心组件

### 2.1 输入层（投递箱）

| 组件 | 路径 | 功能 |
|------|------|------|
| `raw/clippings/` | 网页剪藏 | Web Clipper 收藏的网页 |
| `raw/articles/` | 文章文档 | PDF/Word/PPT 等资料 |
| `templates/` | AI模板 | AI生成物输出模板（人工维护） |

**工作流程：**
```
用户复制文件到 _inbox/ → 运行 process 命令 → 
系统自动分类/提取元数据 → 生成带 AI 分析的笔记 → 
移动到 achievements/ → 归档原文件
```

### 2.2 处理层（脚本引擎）

| 脚本 | 功能 | 触发方式 |
|------|------|----------|
| `obsidian.py` | 统一入口 | 手动/定时 |
| `fetch_policy.py` | 政策简报获取 | 手动/每日自动 |
| `auto_evolve.py` | 知识演化 | 定时/手动 |
| `weekly_report.py` | 周报生成 | 每周定时 |
| `generate_graph.py` | 知识图谱 | 手动/定时 |

### 2.3 AI 层（智能处理）

```
用户输入 → 本地向量检索 → 提取相关文档 → 
构建上下文 → 调用 Claude API → 生成回答 → 
返回结果并引用来源
```

### 2.4 输出层（成果库）

```
achievements/
├── weekly/      # 周报
├── research/    # 研究文档
├── design/      # 设计文档
├── code/        # 代码片段
└── documents/   # 一般文档
```

## 三、运行机制详解

### 3.1 文件处理流程

```
┌────────────────┐     ┌────────────────┐     ┌────────────────┐
│  投递箱        │────▶│  AI 分析       │────▶│  成果库        │
│  (_inbox/)     │     │  detect_category()│  │  (achievements/)│
└────────────────┘     └────────────────┘     └────────────────┘
       │                      │                       │
       ▼                      ▼                       ▼
  用户投放文件           自动分类                    自动建档
  - PDF                 - 周报                      - Front matter
  - Word                - 研究                      - 元数据
  - Markdown            - 设计                      - 附件处理
  - 其他                - 代码                      - 归档
```

**具体步骤：**
1. **检测分类** - 根据文件名关键词判断类型
2. **提取元数据** - 标题、日期、来源
3. **生成笔记** - 添加 AI 提示模板
4. **处理附件** - 复制到 .attachments 目录
5. **归档** - 移动原文件到 _archive/

### 3.2 网页采集流程

```
┌────────────────┐     ┌────────────────┐     ┌────────────────┐
│  添加网页      │────▶│  获取内容      │────▶│  生成笔记      │
│ web add <URL>  │     │ requests + bs4 │     │ 自动摘要       │
└────────────────┘     └────────────────┘     └────────────────┘
       │                      │                       │
       ▼                      ▼                       ▼
  JSON 任务队列           HTML 解析                Markdown
  - URL                  - 去除脚本样式            - 内容摘要
  - 标题                 - 提取正文                - 元数据
  - 标签                 - 截取前 2000 字          - 原文链接
```

### 3.3 AI 对话流程

```
┌────────────────┐     ┌────────────────┐     ┌────────────────┐
│  用户提问      │────▶│  向量检索      │────▶│  构建 Prompt   │
│  ask "问题"    │     │  本地搜索      │     │  上下文组装    │
└────────────────┘     └────────────────┘     └────────────────┘
       │                      │                       │
       ▼                      ▼                       ▼
  自然语言              相似度匹配             相关文档+问题
                        top-k                   发送给 AI

┌────────────────┐     ┌────────────────┐     ┌────────────────┐
│  AI 生成回答   │────▶│  返回结果      │────▶│  保存记录      │
│  Claude API    │     │  终端输出      │     │  对话历史      │
└────────────────┘     └────────────────┘     └────────────────┘
       │                      │                       │
       ▼                      ▼                       ▼
  带引用的回答           直接展示                 可追溯
```

### 3.4 GitHub Actions 自动化

| Workflow | 触发条件 | 执行内容 |
|----------|----------|----------|
| `daily-policy.yml` | 每日/手动 | 获取政策简报、生成报告 |
| `auto-compile.yml` | push raw/* | 自动编译知识库 |
| `knowledge-graph.yml` | push | 生成知识图谱、索引更新 |
| `auto-evolve.yml` | 每周 | 知识演化、缺口分析、建议生成 |
| `weekly-report.yml` | 每周五 | 生成周报 |

## 四、产生的效果

### 4.1 自动化效果

| 场景 | 传统方式 | East-lake 方式 |
|------|----------|----------------|
| 添加研究文档 | 手动创建笔记、复制内容、添加标签 | 放入投递箱 → 自动处理 |
| 收集网页 | 手动复制粘贴 | `web add URL` → 自动采集摘要 |
| 知识问答 | 人工搜索+整理 | `ask "问题"` → AI 自动回答 |
| 知识演化 | 手动维护索引 | 定时自动执行 + 建议生成 |
| 文档分类 | 手动打标签 | AI 自动分析+分类 |

### 4.2 效率提升

| 操作 | 传统耗时 | East-lake 耗时 | 提升 |
|------|----------|----------------|------|
| 添加文档 | 5-10 分钟 | 30 秒 | 10x |
| 网页采集 | 10 分钟 | 10 秒 | 60x |
| 知识问答 | 30 分钟 | 5 秒 | 360x |
| 批量处理 | 手动逐个 | 批量自动 | 依数量 |

### 4.3 质量保证

- **一致性**：所有文档使用统一模板和元数据格式
- **可追溯**：每篇笔记记录来源、处理时间、状态
- **可发现**：自动生成的知识图谱便于关联发现
- **可扩展**：API 服务支持外部系统集成

## 五、使用场景示例

### 场景 1：团队成员提交周报

```bash
# 1. 成员将周报文件放入投递箱
cp weekly-report.pdf ~/East-lake/_inbox/

# 2. 系统自动处理（手动或定时）
python scripts/obsidian.py achieve process

# 3. 结果
# - 自动识别为 weekly 类别
# - 生成 achievements/weekly/2026-05-06-weekly-report.md
# - 包含模板化的笔记结构
# - 附件保存在同目录 .attachments/
```

### 场景 2：采集研究文章

```bash
# 1. 添加网页
python scripts/obsidian.py web add "https://example.com/research"

# 2. 处理采集
python scripts/obsidian.py web process

# 3. 结果
# - 生成 achievements/research/2026-05-06-xxx.md
# - 包含内容摘要、原文链接、标签
```

### 场景 3：知识问答

```bash
# 1. 提问
python scripts/obsidian.py ask "我们团队今年的研究重点是什么？"

# 2. 系统流程
# - 检索所有相关文档
# - 提取关键信息
# - 构建上下文
# - 调用 AI 生成回答

# 3. 返回结果（带来源引用）
```

## 六、配置说明

### 6.1 必需配置

```bash
# AI 功能（可选）
export ANTHROPIC_API_KEY="your-api-key"

export GITHUB_TOKEN="your-github-token"
```

```json
{
  "model": "claude-sonnet-4-20250514",

  "temperature": 0.7,
  "search_top_k": 5

```

## 七、故障排查

### 7.1 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 投递箱文件未处理 | 脚本未运行 | 运行 `achieve process` |
| AI 对话失败 | 无 API Key | 设置 ANTHROPIC_API_KEY |
| 网页采集失败 | 网络问题 | 检查网络连接 |
| API 服务无法启动 | 端口占用 | 检查 8000 端口 |

### 7.2 诊断命令

```bash
# 系统诊断
python scripts/diagnose.py

# 查看状态
python scripts/obsidian.py status
```
